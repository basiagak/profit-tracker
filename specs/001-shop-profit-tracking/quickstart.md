# Quickstart: Shop Profit Tracking

Validation guide for proving the feature works end-to-end. Not implementation code — see `data-model.md` for schema, `contracts/` for the exact command/route surface, and `tasks.md` (generated by `/speckit-tasks`) for build steps.

## Prerequisites

- Go 1.23+
- PostgreSQL 15+ running locally (or via Docker) and reachable via a DSN
- A Telegram bot token (create via [@BotFather](https://t.me/BotFather)) for local testing of `/start`, `/ingredient`, `/item`, `/purchase`, `/sale`, `/report`
- `golang-migrate` CLI (or the project's wrapped `go run` migration command, once implemented) to apply `migrations/`
- `docker` available locally for `testcontainers-go` integration tests
- `curl` (or similar) for exercising the JSON API directly — the API has no HTML pages of its own (see `contracts/web-api.yaml`), so manual dashboard validation below is done via HTTP requests, not a browser

## Setup

```bash
# 1. Configure environment
export DATABASE_URL="postgres://user:pass@localhost:5432/profit_tracker?sslmode=disable"
export TELEGRAM_BOT_TOKEN="<token from BotFather>"
export SESSION_SECRET="<random 32+ byte secret>"
export HTTP_ADDR=":8080"

# 2. Apply schema migrations
migrate -database "$DATABASE_URL" -path migrations up

# 3. Run the service (starts both the Telegram long-polling bot and the JSON HTTP API)
go run ./cmd/server
```

API available at `http://localhost:8080/api/*` (see `contracts/web-api.yaml` for the full route list — it returns JSON only, no HTML); bot reachable via the Telegram account tied to `TELEGRAM_BOT_TOKEN`.

Since `POST /api/auth/telegram` expects a Telegram Login Widget payload (which needs a real widget embedded in an out-of-scope client to produce), local API testing can instead authenticate by computing a valid `hash` by hand: HMAC-SHA256 of the sorted `key=value` fields (excluding `hash` itself) joined by `\n`, keyed by `SHA256(TELEGRAM_BOT_TOKEN)`, per Telegram's [login widget verification algorithm](https://core.telegram.org/widgets/login#checking-authorization) — a one-off script (not a shipped project component) is enough to produce a valid test payload.

## Manual validation scenarios

Each scenario maps to an acceptance scenario in `spec.md`; run them in order against a fresh database (or a fresh shop/user) since later stories depend on catalog/purchase data from earlier ones.

### 1. Set up shop catalog (User Story 1)

1. In Telegram, send `/start` to the bot → confirms shop auto-provisioned.
2. `/ingredient add Flour kg` → appears via `/ingredient list`.
3. `/item add "Bread Loaf" 5.00` → appears via `/item list`.
4. `/item recipe "Bread Loaf" Flour 0.5` → recipe set.
5. `/ingredient archive Flour` → still shows in `/item recipe` history for "Bread Loaf" but no longer offered when starting a *new* recipe/purchase.
6. From a second Telegram account, attempt `/ingredient list` → sees only that account's own (empty) catalog, never the first account's "Flour"/"Bread Loaf".

**Expected**: catalog entities scoped per shop; archive hides from new-entry flows but preserves history; cross-shop access denied (FR-002, FR-005).

### 2. Record purchases and track cost (User Story 2)

1. `/purchase Flour 10 25.00 2026-07-01` → reply shows `current_unit_cost = 2.5000` (25.00 / 10).
2. `/purchase Flour 5 15.00 2026-07-05` → reply shows updated `current_unit_cost = 3.0000` (15.00 / 5, the *most recent* purchase per FR-010).
3. `curl -b cookies.txt "http://localhost:8080/api/purchases?ingredient_id=<flour-id>"` → both purchase rows still show their original recorded `quantity`/`unit_cost`/`purchase_date`, unaffected by step 2.

**Expected**: current unit cost always reflects only the latest purchase; past purchase rows never mutate (FR-009–FR-012).

### 3. Record a sale with locked-in cost (User Story 3)

1. With "Bread Loaf" recipe = 0.5kg Flour at `current_unit_cost = 3.0000`, send `/sale "Bread Loaf" 2`.
2. Reply should show `unit_production_cost = 1.5000` per loaf (0.5 × 3.0000) and `unit_price = 5.00`, confirmed within a few seconds.
3. Now record another purchase that changes Flour's `current_unit_cost` (e.g., `/purchase Flour 10 50.00`) and edit the recipe (`/item recipe "Bread Loaf" Flour 0.75`).
4. Re-query the original sale (via `curl -b cookies.txt "http://localhost:8080/api/sales?from=...&to=..."` or a report for that date) → its stored `unit_production_cost` is still `"1.5000"`, unchanged by step 3.

**Expected**: production cost (and price) snapshot is permanent; later cost/recipe edits never retroactively change a past sale (FR-008, FR-014, FR-015).

### 4. View profit dashboard (User Story 4)

1. Seed a few more sales across at least two different dates.
2. `curl -b cookies.txt "http://localhost:8080/api/reports/summary?from=<start>&to=<end>"` covering all seeded sales → totals for `revenue`, `cogs`, `gross_profit` match manual sums of the seeded `sale_items` snapshots.
3. Narrow the date range to exclude some sales → totals and `trend` array update to reflect only the included sales.
4. Re-run a `/report` for a past date range already viewed once, after further ingredient-cost/recipe changes → figures are byte-for-byte identical to the first time that range was viewed (and identical to what `GET /api/reports/summary` returns for the same range).

**Expected**: dashboard and `/report` agree; changing the date range changes only which sales are included; past-range figures are 0%-drift stable (FR-016–FR-018, SC-006).

### Edge cases to spot-check

- `/sale` for an item whose recipe references an ingredient with no purchases yet → rejected with a clear error, no sale recorded.
- `/purchase Flour -1 10` and `/purchase Flour 1 -10` → both rejected, no write.
- A malformed command, e.g. `/purchase Flour` (missing quantity/cost) → bot replies with usage help, no write.
- `curl -b cookies.txt "http://localhost:8080/api/reports/summary?from=2099-01-01&to=2099-01-31"` (a range with zero sales) → `200` with zero totals (`"revenue": "0.00"`, etc.) and an empty `trend` array, no error.
- A request to any `/api/*` route without a valid session cookie → `401 { "error": "unauthenticated" }`, JSON body, no HTML redirect.
